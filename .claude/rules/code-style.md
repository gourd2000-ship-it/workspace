#일반 원칙

- 가독성 우선: 코드는 사람과 AI 에이전트 모두가 읽기 쉬워야 합니다. 복잡한 한 줄짜리 코드보다는 명확한 여러 줄의 코드를 선호합니다.

- 함수형 프로그래밍 지향: 가능한 한 순수 함수를 작성하고 상태 변경을 최소화합니다.

- 모듈화: 하나의 함수는 하나의 기능만 수행하도록 작게 유지합니다.

#언어별 명명 규칙

- Python: 변수와 함수 이름은 snake_case를 사용하고, 클래스 이름은 PascalCase를 사용합니다. 상수는 모두 대문자인 SCREAMING_SNAKE_CASE로 작성합니다.

- JavaScript 및 Google Apps Script: 변수와 함수 이름은 camelCase를 사용하고, 클래스는 PascalCase를 사용합니다.

- 전역 변수 지양: 전역 범위의 변수 사용을 피하고 필요한 경우 설정 객체나 환경 변수로 관리합니다.

#주석 및 문서화 규정

- JSDoc/Docstring: 모든 함수 상단에는 매개변수의 타입과 반환값, 기능 설명을 포함하는 표준 문서화 주석을 작성합니다.

- 설명적 주석: 왜(Why) 이 로직이 필요한지에 집중하여 주석을 답니다. 어떻게(How)는 코드 자체로 드러나야 합니다.

- 한국어 사용: 주석과 에러 메시지는 한국어로 작성하여 가독성을 높입니다.

#에러 처리 및 디버깅

- Try-Catch 활용: 외부 API(LLM API, n8n 웹훅 등) 호출 시에는 반드시 에러 처리 구문을 포함합니다.

- 의미 있는 에러 메시지: 단순히 에러가 발생했다는 메시지 대신, 어떤 단계에서 어떤 이유로 실패했는지 구체적으로 출력합니다.

- 로깅: 중요 단계마다 console.log() 또는 logging 모듈을 사용하여 실행 흐름을 추적할 수 있게 합니다.
